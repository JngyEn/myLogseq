		- 具体考点：
			- **一、 活动图 (Activity Diagram)**   -   **必考，重点！**
				- 这是必考题，而且是序列图的基础，务必掌握扎实。
				- **核心要求**:
					- **决策节点 (Decision Node)** : 菱形判断框的每个分支后面，必须至少连接一个**活动 (Activity)** ，不能只有`[守护条件]`然后就直接合并。
						- **正确示例**: `判断 -> [条件A] -> 活动A -> 合并`
						- **错误示例**: `判断 -> [条件A] -> 合并`
					- **分叉/汇合 (Fork/Join)** : `Fork`节点（黑棒）分出去几条并发流程，对应的`Join`节点就必须有几条流程汇合进来。数量必须严格相等。
					- **可能出现**: 可能会考到 "接受事件动作 (Accept Event Action)" 或 "接受时间事件动作 (Accept Time Event Action)"。这通常表示系统在等待某个外部信号或等待一段时间。
				- **简化要求 (不考的内容)** :
					- 对象节点 (Object Node)
					- 参数 (Parameter)
					- 异常处理器 (Exception Handler)
					- 可中断活动区域 (Interruptible Activity Region)
					- 结构化活动 (Structured Activity)
					- 避免使用扩展区域 (Expansion Region)，即循环结构。
				- **备考建议**:
					- 重点练习如何正确使用**决策节点**和**分叉/汇合**。这是最容易出错的地方。
					- 找一些业务流程的例子（如网上购物、注册登录、请假审批等），自己动手画活动图，画完后对照要求检查。
			- **二、 序列图 (Sequence Diagram)**   -   **必考，重点！**
				- 这道题和活动图紧密相关，逻辑要保持一致。
				- **核心要求**:
					- **基于活动图**: 序列图的逻辑流程需要根据题目给出的活动图来绘制。确保消息的顺序和活动图中的活动顺序一致。
					- **生命线 (Lifeline)** : 题目会提供给你需要用到的生命线（即参与交互的对象）。你不需要自己设计有哪些对象。
					- **操作参数**: 在绘制消息时，操作/方法调用需要写清楚**参数**。
						- **正确示例**: `登录(用户名, 密码)`
						- **错误示例**: `登录()`
				- **备考建议**:
					- 练习“看活动图画序列图”。找一个你画好的活动图，思考每个“活动”是哪个“对象”执行的，对象之间如何通过“消息”来触发下一个活动。
					- 记住，序列图强调的是**对象之间**的时间交互顺序。
			- **三、 状态机图 (State Machine Diagram)**   -   **可能在选考题中**
			- **核心要求**:
				- **条件/触发器 (Condition/Trigger)** : 重点掌握状态转换的格式。一个完整的转换通常表示为：`触发事件 [守护条件] / 动作` (Trigger [Guard] / Activity)。
				- 考试可能会给你一个对象的生命周期描述（比如订单有“待支付”、“已支付”、“已发货”、“已完成”、“已取消”等状态），让你画出它的状态机图。
			- **备考建议**:
				- 理解**状态**、**转换**、**事件**、**条件**和**动作**这几个核心概念的区别。
				- 练习识别一个对象在不同条件下会进入哪些不同的状态。
			- **四、 通信图 (Communication Diagram)**   -   **可能在选考题中**
				- 通信图和序列图表达的是同样的信息，只是形式不同。
				- **核心要求**:
					- **操作不能重叠**: 这指的是消息的**编号**。要使用清晰的、带小数点的十进制编号来表示消息的顺序和嵌套关系（例如 `1.`、`1.1.`、`1.2.`、`2.`）。在图上绘制时，确保这些编号和箭头清晰可见，不要画得乱七八糟挤在一起。
				- **备考建议**:
					- 通信图强调的是**对象之间**的组织关系（谁和谁有联系）。
					- 可以练习把你画的序列图转换成通信图。这能帮你更好地理解两者之间的对应关系。
